# REDCap: Multiple Vulnerabilities

## Vendors

* Vanderbilt University Medical Center (VUMC)

## Affected Products

REDCap v15.4.2 and earlier (tested version). The Remote Code Execution bypass affects all versions of REDCap prior to 15.5.34 LTS, 16.0.12 LTS, and 16.1.0 Standard Release.

## Summary

REDCap is vulnerable to the following issues that could lead to remote code execution, SQL injection, or unauthorized access to sensitive data. Most findings require authentication; one is exploitable without authentication:

* Remote Code Execution via LogicParser Bypass
* SQL Injection in AI Features
* Unauthenticated Access to All Survey Files
* Insecure Re-Use of Cryptographic Key

Additionally, the following vulnerabilities could lead to authentication bypass or session compromise:

* Multi-Factor Authentication Bypass
* Reflected Cross-Site Scripting in ehr_settings.php

A bypass of the original Remote Code Execution fix was later identified and addressed in a subsequent release.

## Remediation/Mitigation

These issues were patched across multiple REDCap releases:

* The SQL Injection, initial Remote Code Execution, and Reflected Cross-Site Scripting findings were fixed in REDCap 15.0.32 LTS and 15.5.2 Standard Release.
* The cryptographic key reuse and unauthenticated survey file access findings were addressed programmatically in a REDCap release on 2025-09-25.
* The bypass of the Remote Code Execution fix was patched in REDCap 15.5.34 LTS, 16.0.12 LTS, and 16.1.0 Standard Release.

Customers should upgrade to the latest version of REDCap.

For institutions unable to immediately upgrade, the following workarounds reduce exposure:

* Configure REDCap to require multi-factor authentication for users sourcing from all addresses, removing the IP-based MFA bypass path.

## Technical Details

### Remote Code Execution via LogicParser Bypass

The REDCap application contains functionality that allows users to define expressions for the purposes of data filtering and alerting. The expression language implementation is insecure and can be abused to execute arbitrary PHP code on the backend server by any authenticated user.

One feature used in several places throughout the REDCap application is REDCap `logic`. In REDCap, `logic` expressions are user-defined PHP expressions that are restricted to safe operations which can be applied to REDCap data. For example, it is possible for low privileged users to configure `logic` which will trigger alerts to be sent via email or notification when the `logic` expression evaluates to `true`.

Higher privilege users can configure and test `logic` expressions in the `Online Designer` sections of the application.

There are several other places throughout the application where `logic` expressions can be configured and run by low and high privilege users. The following shows an example of a logic expression that bypasses the filters.

```php
logic=<@urlencode>concat("a", "b{$_GET[\'a\'](\'cat /etc/passwd\')}")</@urlencode>
```
*Logic to Disclose /etc/passwd*

Following the initial fix, Atredis Partners identified a bypass that remained exploitable through alternate PHP syntax. While the original payload used the `${}` syntax (which no longer worked after the fix), it was possible for an authenticated user to send logic in the format `$\{}` and achieve arbitrary code or operating system command execution. This bypass was addressed in a subsequent release.

### SQL Injection in AI Features

The REDCap application is vulnerable to SQL injection in the AI data summary feature of the application. An authenticated attacker may be able to exploit this to extract or modify arbitrary data in the backend database resulting in privilege escalation within the application or remote code execution on the REDCap server.

Atredis Partners was able to use the tool `sqlmap` to create a proof of concept that can retrieve arbitrary values from the backend database. For demonstration purposes, Atredis Partners used it to retrieve the database user.

```text

Parameter: report_id (POST)
```

### Unauthenticated Access to All Survey Files

REDCap uses a user defined encryption key called a `salt` in configuration to protect user uploaded files. If the salt is not sufficiently complex it can be anonymously recovered through offline password guessing attacks because SHA1 hashes that have a known plaintext format are exposed by REDCap. It is possible to use the recovered secret salt value to access any document that has been uploaded to a given survey. This attack can be performed without valid authentication credentials to REDCap.

This attack is relatively complex but feasible.

The REDCap `file_download.php` allows unauthenticated access when it is used from the context of a survey. This is to facilitate survey users downloading files such as files they have previously uploaded. Survey hashes should be considered public information; Atredis Partners was able to find several valid values for survey hashes via a Google search.

The `file_download.php` protects access to sensitive files by hashing the file id with a secret value. If the secret value is not sufficiently complex it can be recovered through offline password cracking and arbitrary files can be downloaded.

### Insecure Re-Use of Cryptographic Key

The REDCap application re-uses the same cryptographic key for encryption, decryption, and hashing throughout the application. In many cases, hashes or encrypted values are exposed to users in a known format. Re-use of the key can result in additional exposure of the key and make key recovery via offline password cracking possible if the key is weak.

Atredis Partners noted that the same `salt` value that is used to calculate CSRF token values is used for encryption, decryption, and hashing throughout the application including to protect survey file downloads from unauthorized access.

Different secret keys should be used for different functions; the same key should not be used for hashing as encryption.

### Multi-Factor Authentication Bypass

REDCap instances can be configured to allow users originating from specified network addresses to bypass Multi-Factor Authentication (MFA). Multi-factor authentication requirements can be bypassed by spoofing the user's source IP address in an `X-Forwarded-For` HTTP header.

Due to the way REDCap processes client source addresses for HTTP requests, it is possible for attackers to spoof their source address by using the `X-Forwarded-For` header.

Atredis Partners confirmed dynamically that it was possible to bypass MFA controls by setting the `X-Forwarded-For` header value to `10.0.0.1`.

As a workaround prior to upgrading, REDCap instances should be configured to require multi-factor authentication for users sourcing from all addresses.

### Reflected Cross-Site Scripting in ehr_settings.php

The REDCap portal returns user-supplied data in the server's response without proper encoding, resulting in Cross-Site Scripting (XSS). An attacker that can coerce a user to navigate to a malicious URL could perform actions on the REDCap platform on behalf of that user. Due to the vulnerable endpoint requiring an HTTP POST request, user interaction would be required.

The following request to `ehr_settings.php` reflects script in the POST body.

```http
POST /redcap_v15.4.2/ControlCenter/ehr_settings.php HTTP/1.1
Host: <redacted>
Content-Type: application/x-www-form-urlencoded
Content-Length: 47

atredis=<script>alert(document.domain)</script>

HTTP/1.1 200 OK
Content-Type: text/html; charset=UTF-8

<pre>Array
(
    [atredis] => <script>alert(document.domain)</script>
)
</pre>
```
*Cross-Site Scripting Payload Returned*

Due to the lack of Cross Origin Resource Sharing (CORS) headers and Cross-Site Request Forgery (CSRF) protections on this endpoint, it is possible to make this HTTP POST request from another domain.
